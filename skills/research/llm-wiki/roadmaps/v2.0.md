# LLM Wiki Skill Roadmap v2.0

## Purpose

`v2.0` is the planned upgrade path once the `v1.0` markdown-first system starts to feel stretched by scale, query complexity, and maintenance overhead.

This version is not a rejection of markdown. It is a structured enhancement of the same system.

The core idea is:

- markdown remains the primary human-facing surface
- the CLI becomes a real operating layer
- SQLite is introduced as an optional assistant for indexing, joins, and validation

## When v2.0 Is Needed

Move to `v2.0` when the wiki repeatedly shows signals such as:

- the same concept or artifact keeps appearing under multiple names
- it becomes unclear which page is canonical
- topic maps help, but search recall is still weak
- lint wants information that is awkward to compute from raw file scans alone
- queries increasingly need combinations like source plus time plus artifact plus domain
- maintenance work begins to feel like database work done manually

`v2.0` is driven by operating pain, not by page count alone.

## Core Objective

Keep the user experience of `v1.0`, but make the system easier to query, maintain, and extend.

`v2.0` should:

- preserve markdown as the editable knowledge surface
- turn the CLI into a stable maintenance and indexing interface
- introduce SQLite as a helper layer rather than a replacement truth source
- make registries easier to rebuild, validate, and query
- improve recall and navigation without forcing a custom GUI

## Architecture Shift

`v2.0` keeps the four-layer structure but strengthens two of the layers:

1. `pages/`
   Still the main readable knowledge surface.

2. `meta/`
   Becomes more consistent and easier to generate from source files.

3. `tools/wiki-cli/`
   Evolves from lint-only to a broader operational interface.

4. SQLite index
   Added as an optional helper layer for structured queries and fast checks.

The key rule is unchanged:

Markdown remains the primary interface for humans.

## Role Of SQLite

SQLite in `v2.0` is an indexing and validation assistant, not the main truth source.

It should be used for:

- source registry joins
- page registry joins
- alias lookup
- backlink and reference indexing
- duplicate candidate analysis
- stats and reporting
- recall support for query preparation

It should not become the only place where knowledge exists.

If markdown disappears, the system has drifted too far from the intended design.

## CLI Expansion

`v2.0` should make the CLI a real maintenance surface.

Probable commands:

- `wiki_cli lint`
- `wiki rebuild-registry`
- `wiki stats`
- `wiki resolve-aliases`
- `wiki map`
- `wiki query-prep`

`wiki ingest` may remain partially skill-driven in `v2.0`, but the CLI should provide reliable deterministic helpers so the skill does less fragile bookkeeping itself.

## Expected CLI Responsibilities

By `v2.0`, the CLI should own more of the deterministic system behavior:

- validating structure
- rebuilding metadata registries
- scanning wikilinks and references
- generating stats and change summaries
- suggesting duplicate candidates
- checking coverage gaps
- supporting maintenance workflows with structured outputs

The CLI should still avoid taking over semantic writing decisions.

## Metadata Maturity

`v2.0` expects the metadata model from `v1.0` to be stable enough to support extraction into SQLite tables.

Likely indexed objects:

- sources
- pages
- aliases
- page links
- source refs
- domains
- topic membership

Potential later additions:

- lightweight claims
- evidence references
- staleness indicators

These additions should be evolutionary, not disruptive.

## Query Improvements

`v2.0` should improve the agent query workflow without replacing wiki pages.

Desired query behavior:

1. ask the index which pages and sources are likely relevant
2. read the corresponding markdown pages
3. synthesize from markdown, not only from rows
4. optionally record the result back into `question` or `synthesis` pages

This preserves the wiki's role as a compiled memory layer while making recall more reliable.

## Topic Maps And Navigation

`v2.0` should strengthen topic-map maintenance rather than bypassing it.

Expected improvements:

- CLI-assisted reports showing underlinked themes
- better visibility into which sources and pages belong to a topic
- registry support for canonical topic membership
- easier detection of fragmented topic coverage

Topic maps remain the human-readable navigation layer even if the index becomes much stronger.

## Meetings In v2.0

Meetings remain fully in scope.

`v2.0` should make it easier to:

- index meeting-derived source notes
- connect meetings to related papers and artifacts
- track recurring research questions across meetings
- surface meeting notes that should feed into synthesis updates

Meetings are especially valuable because they often capture belief changes earlier than formal documents do.

## Weekly Maintenance In v2.0

The weekly maintenance loop should become more informative and less manual.

A mature weekly flow could look like:

1. detect new or changed sources
2. run `wiki_cli lint`
3. rebuild registries or indexes if needed
4. produce a summary of structural issues and recommended semantic maintenance
5. optionally trigger targeted LLM maintenance tasks

The weekly loop should still avoid large automatic rewrites.

## Non-Goals

`v2.0` should still avoid:

- replacing markdown with a database UI
- forcing a GUI beyond Obsidian or the user's editor of choice
- turning the skill into a full application framework
- moving semantic authorship into deterministic tooling

This phase is about support infrastructure, not productizing the whole system.

## Exit Criteria

`v2.0` is successful when:

- the CLI is a dependable maintenance surface
- SQLite meaningfully reduces search and validation friction
- registry rebuilds no longer feel manual
- alias and canonical-page management are noticeably better
- the system still feels markdown-first to the user
- the wiki can handle significantly more scale without a full architectural rewrite

## Looking Ahead To v3.0

`v3.0` should only happen if the system starts to need:

- richer structured queries as a routine workflow
- more formal maintenance pipelines
- more explicit claim or evidence modeling
- stronger automation around semantic upkeep

If `v2.0` is designed well, `v3.0` can remain CLI-first and still avoid requiring a custom GUI.
