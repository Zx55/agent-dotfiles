# LLM Wiki Skill Roadmap v1.0

## Purpose

`v1.0` defines the first production-worthy version of the `llm-wiki` skill for a personal AI research workflow.

This version is intentionally not a database-first knowledge system. It is a markdown-first workflow with enough structure to stay maintainable and enough metadata to support future upgrades.

The target user is a research-heavy user who works across papers, deep research sessions, meetings, product exploration, and new domain discovery.

## Core Objective

Build a durable personal research wiki that:

- is readable and editable directly as markdown
- can be browsed comfortably in Obsidian
- can be updated incrementally by agents
- avoids early schema drift through stable IDs and registries
- uses deterministic linting instead of LLM-only hygiene
- remains compatible with a future CLI and SQLite-assisted indexing layer

## Scope

`v1.0` corresponds to the current `Level 1.5` design.

The system should support:

- markdown pages as the main knowledge interface
- raw source capture for papers, articles, meetings, notes, and product explorations
- a small but durable set of page types
- machine-readable metadata under `meta/`
- a local operating guide in `llm-wiki/AGENTS.md`
- a lint-first CLI strategy under `tools/wiki-cli/`

This version should feel solid at small and medium scale, roughly before the wiki truly demands a structured query engine.

## Architecture Summary

The system is split into four layers:

1. `raw/`
   Immutable source material.

2. `pages/`
   Human-readable knowledge pages.

3. `meta/`
   Lightweight machine-readable registries.

4. `tools/wiki-cli/`
   Deterministic tooling, starting with lint.

This version deliberately keeps markdown as the canonical human-facing layer.

## Supported Inputs

`v1.0` should treat the following as first-class inputs:

- papers and paper notes
- articles and curated links
- meeting summaries and transcripts
- deep research outputs
- product and tool exploration notes
- high-value personal notes

Meeting material is in scope. If a meeting produces technical conclusions, new questions, comparative judgments, or research direction changes, it should be eligible for ingest.

## Durable Page Types

`v1.0` standardizes on these page types:

- `source-note`
- `concept`
- `entity`
- `artifact`
- `question`
- `synthesis`
- `topic-map`

This is intentionally broader than Karpathy's original minimal layout because the target workflow includes papers, models, products, frameworks, benchmarks, and meetings.

## Metadata Contract

All durable pages under `pages/` should carry consistent frontmatter with at least:

- `page_id`
- `title`
- `type`
- `status`
- `created_at`
- `updated_at`
- `aliases`
- `domains`
- `source_refs`
- `related_pages`
- `confidence`
- `review_state`

All captured sources should have stable `source_id` values.

Stable IDs are mandatory in `v1.0` to reduce migration pain later.

## Registries

`v1.0` introduces a lightweight metadata layer under `meta/`.

Expected registries:

- `source-registry`
- `entity-registry`
- `artifact-registry`
- `aliases`
- `topic-registry`
- `query-registry`
- `stats`

These registries do not need to be database-backed in `v1.0`. They only need to be structured enough for deterministic tooling and future extraction.

## Skill Responsibilities

The `llm-wiki` skill in `v1.0` should act as a workflow orchestrator.

It should be responsible for:

- deciding when wiki ingest is appropriate
- orienting itself to existing wiki structure
- selecting the right page type to create or update
- preserving cross-links and topic maps
- writing structured markdown output
- keeping updates incremental

It should not try to be the final source of truth for structural integrity. That belongs to tooling.

## CLI Strategy

`v1.0` should be lint-first.

The first CLI priority is `wiki_cli lint`.

Why lint first:

- the rules are deterministic
- the cost is low
- the payoff is immediate
- it enforces discipline early
- it reduces reliance on prompt-only maintenance

`v1.0` does not require a full command suite. It only requires that the system be designed so a CLI can naturally own deterministic checks.

## Initial `wiki_cli lint` Scope

The first lint version should check:

- required directory structure
- frontmatter presence
- required frontmatter fields
- uniqueness of `page_id` and `source_id`
- valid page types
- valid shapes for aliases, domains, source refs, and related pages
- broken wikilinks
- raw sources missing registry coverage
- sources missing a corresponding `source-note`
- orphan pages
- duplicate and near-duplicate naming candidates

This version should report issues clearly and prefer stable machine-readable output in addition to human-readable summaries.

## Query Policy

The wiki is for both humans and agents.

Default query order in `v1.0`:

1. consult existing `question`, `synthesis`, and `topic-map` pages
2. consult relevant `concept`, `artifact`, `entity`, and `source-note` pages
3. use the web only when the question is time-sensitive or the wiki clearly lacks coverage
4. ingest high-value external findings back into the wiki when appropriate

This keeps the wiki useful as a compounding memory layer instead of a passive archive.

## Topic Maps

`topic-map` pages are a first-class mechanism in `v1.0`.

They exist to prevent knowledge fragmentation across folders and page types.

They should be updated when:

- a theme spans papers, products, and meetings
- a concept interacts with multiple artifacts
- the same question appears across multiple sources
- cross-topic relationships would otherwise be easy to miss

## Meetings

Meetings are explicitly in scope in `v1.0`.

A meaningful meeting should usually produce:

- a raw artifact under `raw/meetings/`
- a stable `source_id`
- a `source-note`
- optional updates to `question`, `concept`, `artifact`, or `synthesis` pages

Routine coordination chatter should not expand into the full wiki.

## Maintenance Model

`v1.0` uses a split maintenance model:

- CLI for structural validation
- LLMs for semantic maintenance

Examples of CLI-owned work:

- schema validation
- link checks
- duplicate candidate checks
- registry coverage checks

Examples of LLM-owned work:

- topic-map refreshes
- synthesis updates
- deciding whether multiple pages should be merged
- extracting durable conclusions from meetings

## Weekly Maintenance

`v1.0` should support a lightweight weekly maintenance loop.

Recommended behavior:

1. detect whether new sources were added during the week
2. run `wiki_cli lint`
3. summarize the current health of the wiki
4. only trigger deeper maintenance if there were additions or structural problems

This keeps the system healthy without causing constant churn.

## Non-Goals

`v1.0` should not attempt to do the following:

- become a database-first app
- require SQLite for normal use
- implement a full CLI suite from day one
- solve semantic contradiction detection perfectly
- introduce a custom GUI
- optimize for multi-user collaboration

Obsidian remains the preferred human reading surface. The wiki itself remains the primary artifact.

## Exit Criteria

`v1.0` is successful when:

- the wiki structure is stable and documented
- `llm-wiki/AGENTS.md` is sufficient to govern work inside the wiki
- the future skill can be written against a clear workflow contract
- meeting material fits naturally into the model
- `wiki_cli lint` can validate the most important structural invariants
- the design is ready for real use before a database or larger tooling stack exists

## Upgrade Trigger For v2.0

The project should move toward `v2.0` when one or more of these become persistent:

- search and navigation friction grow despite topic maps
- alias drift and duplicate pages become frequent
- registry maintenance becomes too manual
- the wiki starts behaving like a database without actually having one
- compositional queries become important to daily usage

At that point the project should add a more capable CLI and an optional SQLite-assisted indexing layer.
