# Page Archetypes

Use this guide to translate a chosen `DESIGN.md` into a page structure that matches the user's actual task.

The goal is not to copy a source site one-to-one. The goal is to keep the spirit of the reference while producing a structure that fits the content.

## Phone Or Hardware Promo Page

Recommended references:

- Apple
- BMW
- SpaceX

Typical structure:

1. hero with product name, one clear promise, and the primary visual
2. key feature sequence with one major idea per section
3. material, display, camera, battery, performance, or ecosystem callouts
4. comparison or specs summary
5. closing CTA

What to borrow:

- cinematic section pacing
- product-first composition
- restrained text density
- large visual transitions

What matters most:

- strong hero hierarchy
- section rhythm
- mobile stacking that still feels premium rather than crowded

## SaaS Landing Page

Recommended references:

- Stripe
- Vercel
- Framer

Typical structure:

1. hero with problem, value proposition, and primary CTA
2. social proof or credibility bar
3. feature grid or workflow explanation
4. product screenshot band
5. integrations, platform, or team collaboration section
6. secondary CTA and footer

What to borrow:

- visual system for trust and clarity
- concise CTA treatment
- card design and spacing
- polished motion or screenshot framing

What matters most:

- fast comprehension above the fold
- clear hierarchy between headline, support text, and actions
- responsive behavior that preserves CTA clarity

## Marketing Homepage

Recommended references:

- Stripe
- Apple
- Vercel
- Notion

Typical structure:

1. hero
2. positioning statement
3. core feature or value sections
4. proof points or case studies
5. product details
6. CTA close

What to borrow:

- pacing
- contrast model
- typography scale
- component feel

What matters most:

- one clear narrative
- not mixing too many visual languages
- keeping section transitions intentional

## Docs Entry Page

Recommended references:

- Mintlify
- Notion
- MongoDB

Typical structure:

1. compact hero with search or primary navigation
2. quickstart cards
3. key documentation categories
4. code example or getting started snippet
5. secondary links for guides, API, and examples

What to borrow:

- calm typography
- navigational clarity
- content density rules
- support surfaces and panels

What matters most:

- readability
- scanning behavior
- avoiding decorative noise that hurts docs usability

## Dashboard

Recommended references:

- Linear
- Sentry
- Raycast

Typical structure:

1. top navigation and status summary
2. primary data panel
3. supporting widgets or tables
4. filters, actions, or recent activity

What to borrow:

- contrast balance
- panel geometry
- subtle status color use
- dense but controlled spacing

What matters most:

- legibility under data density
- interaction affordance clarity
- dark mode discipline if the reference is dark

## Pricing Or Comparison Page

Recommended references:

- Stripe
- Vercel
- Notion

Typical structure:

1. pricing hero and plan chooser
2. tier cards
3. feature comparison matrix
4. FAQ
5. CTA footer

What to borrow:

- pricing card emphasis
- typography hierarchy for plan names and prices
- comparison table rules
- trust-building whitespace

What matters most:

- plan differentiation
- table readability
- clean mobile fallbacks for wide comparisons

## Mobile Adaptation Rule

For every archetype:

- preserve hierarchy first
- preserve rhythm second
- preserve exact desktop composition last

On mobile, keep the same narrative and spacing character even if layout geometry changes substantially.
